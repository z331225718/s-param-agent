解决了什么问题

传统 RF 软件（ADS、VNA 上位机、MATLAB RF Toolbox）的痛点

🖱️

操作繁琐

菜单→对话框→选参数→调坐标轴→导出。一个 S21 dB 图要点十几下。对话式：一句话出图。

📖

学习成本高

每个软件有自己的一套操作逻辑和脚本语言（ADS AEL、MATLAB）。本工具：说人话即可。

🔄

批量处理困难

传统软件处理多文件对比、级联分析需要大量重复操作。Agent 自动理解意图，一次处理多个 S 参数文件。

📊

图表不交互

ADS 和 VNA 导出的是静态图片。本项目输出 Plotly 交互式图表：缩放、悬浮看值、框选、双击重置。

🧠

API 记不住

skrf 有 200+ 个方法、plotly 有上千个参数。本项目的知识图谱 + 自进化机制让 Agent 替你记住正确 API。

🔒

代码不安全

LLM 直接生成代码可能包含危险调用。本项目有 AST 白名单校验 + subprocess 沙箱，os.system/eval 等全部拦截。

三大核心机制

不是简单的「LLM 生成代码」，而是三层防线确保正确性

01

知识图谱纠错

876 节点 + 494 条类型化关系边。当 LLM 用错 API（如 update_xaxis），图谱沿 mistaken_for 边找到正确 API（update_xaxes / update_layout(xaxis_type=)），并附带参数列表和父类信息，一次性纠正。

02

自进化教训库

每次纠错成功后，自动提取「错误模式 → 正确 API」规则，去重加权后持久化到 lessons.json。同一错误再犯时 count++，排序更靠前。下次请求时，累积的教训注入 system prompt，类似 Few-Shot 记忆。

03

三层安全防线

① System Prompt 铁律区（频率必须转 GHz、禁止 SI 前缀、必须科学计数法）
② AST 白名单校验（只允许 skrf/plotly/numpy 的 import，拦截 eval/os/subprocess）
③ subprocess 沙箱执行（独立进程 + 15s 超时 + fig.show() 物理删除）

对话示例

用户说什么，Agent 生成什么代码

用户输入
「读取 filter.s2p，画 S11 和 S21 的 dB 图，左右两个纵轴」
Agent 生成
import skrf as rf import numpy as np import plotly.graph_objects as go from plotly.subplots import make_subplots ntwk = rf.Network('filter.s2p') freq_ghz = ntwk.f / 1e9 # 铁律：频率转 GHz fig = make_subplots(specs=[[{"secondary_y": True}]]) fig.add_trace(go.Scatter(x=freq_ghz, y=ntwk.s_db[:,0,0], mode='lines', name='S11'), secondary_y=False) fig.add_trace(go.Scatter(x=freq_ghz, y=ntwk.s_db[:,1,0], mode='lines', name='S21'), secondary_y=True) fig.update_layout(xaxis_title='Frequency (GHz)', yaxis_title='S11 (dB)', template='plotly_white') fig.update_xaxes(exponentformat='power', showexponent='all') fig.update_yaxes(exponentformat='power', showexponent='all')

下一步：多 S 参数输入

当前支持单文件操作。下一步的核心升级方向：

📂

批量文件加载

拖拽或通配符加载多个 .sNp 文件，自动识别端口数和频率范围，统一或独立坐标系展示。

🔗

智能级联与拓扑

用户描述「LNA → BPF → Amplifier」链路，Agent 自动理解拓扑并计算级联 S 参数。

📊

多文件对比视图

同一参数（如 S21）的多次测量结果叠加对比，自动标注差异和统计信息。

🧮

去嵌与校准

自动识别夹具文件，执行 SOLT/TRL 校准和去嵌操作，支持多端口误差模型。
